# MCP Tools

MCP-сервер dev-workflow предоставляет 26 tools. Конфигурируется в `.mcp.json` при `dev-workflow init`.

## Vault

| Tool | Параметры | Описание |
|------|-----------|----------|
| `vault_status` | -- | Полное состояние vault одним вызовом |
| `vault_read` | `section: stack\|conventions\|knowledge\|gameplan` или `knowledge:<sub>` (sub ∈ architecture, gotchas, security, patterns, engram) | Прочитать секцию vault или sub-секцию knowledge.md |
| `vault_search` | `query: string` | Поиск по всем vault-файлам |
| `vault_record` | `type: adr\|bug\|debt`, `title`, `content` | Создать запись |
| `vault_knowledge` | `section: string`, `content: string` | Дописать в knowledge.md |
| `vault_pattern` | `section?: string`, `content: string` | Дописать pattern-bullet в conventions.md (по умолчанию section = "Patterns") |

### vault_status

Агрегированное состояние проекта — одним вызовом вместо 4+ отдельных reads:

```json
{
  "project": "my-project",
  "branch": "feature/auth",
  "sections": {
    "stack": { "filled": true, "lines": 45 },
    "conventions": { "filled": true, "lines": 82 },
    "knowledge": { "filled": false, "lines": 6 },
    "gameplan": { "filled": true, "lines": 38 }
  },
  "tasks": { "total": 5, "pending": 2, "in-progress": 1, "done": 2 },
  "workflow": null,
  "intelligence": { "patterns": 47, "edges": 23 }
}
```

Используется в `/vault:from-spec` Step 2, `/session:resume`, `/workflow:dev` Step 1.

## Intelligence

| Tool | Параметры | Описание |
|------|-----------|----------|
| `intelligence_query` | `branch?`, `task?`, `limit?` | Поиск паттернов с scoring |

Возвращает массив паттернов, ранжированных по трём критериям:
- **Recency** — давность последнего доступа (half-life 7 дней)
- **Frequency** — частота использования
- **Context match** — совпадение с текущей веткой и задачей

```json
[
  { "id": "file:src/auth.ts", "category": "file", "content": "...", "score": 0.87 },
  { "id": "session:2026-04-01", "category": "session", "content": "...", "score": 0.62 }
]
```

## Memory (Engram proxy)

| Tool | Параметры | Описание |
|------|-----------|----------|
| `memory_search` | `query`, `limit?`, `tags?` | Поиск Engram воспоминаний с auto-decoration (step/branch/run/task теги активного pipeline автоматически добавляются к фильтру) |
| `memory_store` | `context`, `action`, `result`, `type`, `tags?` | Сохранить воспоминание с auto-decoration. Engram раскладывает контент на context/action/result. Ошибки daemon (Voyage 403, offline, и т. д.) поднимаются вызывающему агенту через JSON-RPC `isError: true` |
| `memory_judge` | `memory_id`, `score: 0.0–1.0`, `explanation?` | Оценить полезность воспоминания. Питает Q-learning router для адаптивного отбора контекста |
| `step_complete` | `stepName`, `runId?`, `beforeSearchMemoryIds`, `output` | Финализирует шаг pipeline: парсит `## Engram Feedback` из вывода агента, применяет judgments (silent fail-safe, JUDGE_CAP=20), эмитит antipattern-observability. БЕЗ blanket fallback — id без feedback попадают в `fallbackIds` |

### Auto-decoration

В отличие от прямого `mcp__engram__memory_*`, прокси-обёртки автоматически добавляют теги активного pipeline-контекста: `step:<имя шага>`, `branch:<текущая ветка>`, `run:<run-id>`, `task:<task-id>`. Агенты могут фильтровать воспоминания по фазе исполнения без ручного тегирования. Прямой `mcp__engram__memory_*` остаётся escape hatch для случаев, когда нужен явный контроль над project / tags.

### Asymmetry: strict vs silent (ADR 2026-05-06)

- **User-invoked** `mcp__dev-workflow__memory_store` — throws daemon errors. Если Voyage недоступен или ключ ротирован, агент видит сообщение в tool-result и может реагировать (свитчнуть provider, рестартнуть daemon).
- **Auto-mirror** (`vault_record`/`vault_knowledge`/`vault_pattern`/`EngramBridge.afterStep`) — silent fail-safe. Vault-файл — единственный источник правды; mirror — secondary.

### step_complete

Один вызов в конце каждого шага pipeline вместо ручного цикла `parse_engram_feedback` → N×`memory_judge`. Возвращает:

```json
{
  "judgmentsApplied": 3,
  "fallbackIds": ["<uuid>"],
  "antipatternIdsInBefore": ["<uuid>"],
  "antipatternJudgmentDistribution": {
    "0.0-0.2": 1, "0.2-0.4": 0, "0.4-0.6": 0, "0.6-0.8": 0, "0.8-1.0": 0
  }
}
```

Поведенческие инварианты (ADR 2026-05-13):

- **No blanket fallback** — memories without explicit feedback land in `fallbackIds`; orchestrator решает что делать (не применяется автоматический score).
- **antipatternJudgmentDistribution** — buckets ONLY scores чьи id ∈ antipatternIdsInBefore. 5 buckets left-closed right-open (`0.2` → `0.2-0.4`), верхний `0.8-1.0` инклюзивен.
- **UUID validation** — strict regex throw at boundary (защита от инъекций).
- **JUDGE_CAP=20** per call — bound daemon load; лишние feedback-строки сверх 20 игнорируются.
- **engramJudge silent fail-safe** — daemon недоступен ≠ ошибка step_complete; judgmentsApplied отражает сколько вызовов сделано, не сколько долетело до daemon.

## Workflow и агенты

| Tool | Параметры | Описание |
|------|-----------|----------|
| `workflow_status` | `runId?` | Статус текущего/указанного workflow run |
| `workflow_create` | `name`, `yaml` | Сохранить кастомный workflow в `.dev-vault/workflows/` |
| `workflow_start` | `workflowName`, `taskDescription`, `taskId?` | Инициировать workflow run из slash-оркестратора: создать state-файл в `<vault>/workflow-state/runs/run-<12hex>.json`, выставить `ENGRAM_TRACE_FILE` + `ENGRAM_RUN_ID`, вернуть `{runId, traceFilePath}`. |
| `step_start` | `stepName`, `runId?` | Симметричная пара к `step_complete`: обновляет `run.currentStep` в начале шага. Вызывается до `memory_search` в Step X.0 каждого step-файла — чтобы engram-трейсы получали актуальный `step:<name>` auto-tag. |
| `agent_list` | -- | Список агентов с permissions |
| `agent_run` | `agent`, `task` | Генерация промпта агента с vault-контекстом |
| `parse_engram_feedback` | `output: string` | Парсинг блока `## Engram Feedback` из вывода агента (для пайплайнового scoring) |

### workflow_start

Точка входа для slash-оркестратора (`/workflow:dev`, `/workflow:hotfix`, custom). Резолвит workflow (custom YAML → templates → builtins), генерирует `runId` формата `run-<12hex>` (через `crypto.randomUUID`, prefix `run-` обязателен для `engram-stats` фильтра), создаёт `WorkflowRun` с инициализированным `Record<string, StepState>` (все шаги в `status: "pending"`), сохраняет в `<vault>/workflow-state/runs/run-<12hex>.json`, выставляет `ENGRAM_TRACE_FILE` (если не задан) и `ENGRAM_RUN_ID` (если не задан), возвращает `{runId, traceFilePath}`.

**Валидация (на границе MCP, до любых записей):**

- `workflowName` — regex `/^[a-z0-9][a-z0-9_-]{0,63}$/`, иначе **E001**
- `taskDescription` — non-empty после trim, иначе **E002**
- `taskId` (опционально) — regex `/^task-\d{3,}$/`, иначе **E003**
- Неизвестный workflow — **E004**

**Расхождение генераторов runId с `WorkflowEngine.generateRunId()`:** программный путь (`dev-workflow run`) использует `run-<YYYY-MM-DD>-<seq>` (date-seq); MCP-точка использует `run-<12hex>` (prefixed-hex) — concurrent-friendly для параллельных оркестраторов. Два генератора by design.

### step_start

Симметричная пара с `step_complete`. Оркестратор вызывает `step_start` в самом начале каждого шага pipeline (Step X.0 step-файла) — до первого `memory_search`. Эффект: `run.currentStep` обновляется на текущее имя шага, и все последующие engram-операции получают `step:<name>` в auto-tags. Без `step_start` трейсы шага N все ещё несли бы тег шага N-1.

**Валидация:**

- `stepName` — regex `/^[a-z0-9][a-z0-9_-]{0,63}$/`, иначе **E001**
- `runId` (опционально) — regex `/^run-[a-f0-9]{12}$/`, иначе **E002**
- Активный run отсутствует (нет ни `runId`-параметра, ни `ENGRAM_RUN_ID` env) — **E003**
- `runId` валидного формата, но state-файла нет — **E004** (fail-loud, не silent: оркестратор передаёт битый runId — лучше упасть, чем эмитить мемории под несуществующий шаг)

**Резолюция runId** (priority order): explicit `runId` parameter → `process.env.ENGRAM_RUN_ID` (выставлен `workflow_start`) → throw E003. Возвращает `{ok: true}`.

**Review step special-case:** для `review.md` `step_start` вызывается ОДИН РАЗ перед запуском 3 параллельных reviewer-агентов (не 3 раза) — они делят один и тот же `step:review` тег.

## Communication profiles

3 tools для управления активным профилем общения (см. [Communication Profiles](../concepts/communication)).

| Tool | Параметры | Описание |
|------|-----------|----------|
| `profile_get` | -- | Snapshot: `{configured, active, default, effective, available, config}`. Active = из `.profile-state` (или null), default = из yaml `active_profile`, effective = active ?? default. |
| `profile_set` | `name: string` | Validate `name` существует в `communication.yaml profiles` map (через `Object.prototype.hasOwnProperty.call` — защита от prototype pollution) → write `.dev-vault/.profile-state`. |
| `profile_clear` | -- | Удалить `.dev-vault/.profile-state` (no-op если file отсутствует). После clear `getActiveProfile` fallback на yaml-default. |

Slash `/profile` (см. [commands/session](../commands/session#profile)) — thin wrapper над этими tools. Прямой вызов через MCP полезен для CLI-скриптов / automation.

## Где конфигурируется

MCP-сервер настраивается в `.mcp.json` (не в `.claude/settings.json`):

```json
{
  "mcpServers": {
    "dev-workflow": {
      "command": "node",
      "args": ["/absolute/path/to/dist/cli/index.js", "serve"]
    }
  }
}
```

Claude Code автоматически обнаруживает `.mcp.json` в корне проекта.
